# 프로젝트 헌장 (Project Constitution)

이 문서는 복소 좌표계 그래프 시각화 프로젝트의 개발 및 유지보수에 있어 지켜야 할 핵심 원칙과 가치를 정의합니다. 모든 기여자(AI 및 인간)는 이 원칙을 준수해야 합니다.

## 1. 핵심 철학 (Core Philosophy)

### 1.1 시각적 직관 우선 (Visual First)
- **원칙**: 수학적 개념은 시각적으로 즉시 확인될 때 가장 잘 이해됩니다.
- **실행**: 사용자가 수식을 입력하는 즉시 그래프가 반응해야 합니다. 텍스트보다 그래프가 먼저 눈에 들어와야 합니다.

### 1.2 탐험적 학습 (Exploratory Learning)
- **원칙**: 사용자는 수동적인 관찰자가 아니라 능동적인 탐험가여야 합니다.
- **실행**: 정적인 그래프보다는 동적인 슬라이더와 애니메이션을 통해 파라미터 변화가 결과에 미치는 영향을 실시간으로 체험할 수 있어야 합니다.

## 2. 사용자 경험 원칙 (UX Principles)

### 2.1 즉각적인 피드백 (Immediate Feedback)
- 입력 후 그래프 렌더링까지의 지연을 최소화합니다.
- 잘못된 수식 입력 시, 사용자가 좌절하지 않도록 즉각적이고 명확한, 그러나 방해되지 않는 에러 표시를 제공합니다.
- **목표**: 앱 로드 후 5초 이내에 첫 그래프를 그릴 수 있어야 합니다.

### 2.2 자연스러운 입력 (Natural Input)
- 엄격한 문법보다 사용자의 의도를 파악하는 파싱을 지향합니다.
- 예: `2*i` 대신 `2i`와 같은 암시적 곱셈을 적극 지원합니다.

### 2.3 매끄러운 인터랙션 (Fluid Interaction)
- 그래프의 팬(Pan), 줌(Zoom), 슬라이더 조작은 끊김 없이 부드러워야 합니다.
- **목표**: 인터랙션 시 평균 60fps 유지를 지향합니다.

## 3. 기술적 표준 (Technical Standards)

### 3.1 성능 및 최적화 (Performance)
- 수학적 계산이 UI 스레드를 차단(Block)하지 않아야 합니다. 필요한 경우 워커(Worker) 스레드 사용을 고려합니다.
- 렌더링 루프는 효율적이어야 하며, 불필요한 재계산을 피합니다.

### 3.2 견고성 (Robustness)
- **예외 처리**: 0으로 나누기, 무한대 발산 등 수학적 예외 상황에서도 애플리케이션이 멈추거나 깨져서는 안 됩니다. (예: 화면 밖 클리핑 처리)
- **입력 방어**: 예약어 충돌이나 잘못된 변수 할당을 사전에 방지하거나 우아하게 처리합니다.

### 3.3 확장성 (Extensibility)
- 핵심 엔티티(`Expression`, `Variable`, `PlotObject`)는 명확히 분리되어야 하며, 추후 3D 그래프나 다른 수학적 도구로 확장 가능하도록 설계합니다.

## 4. 커뮤니케이션 및 코드 스타일 (Communication & Style)

- **명확성**: 변수명과 함수명은 수학적 의미(예: `realPart`, `imaginaryPart`)를 명확히 담아야 합니다.
- **문서화**: 복잡한 수식 처리 로직이나 렌더링 최적화 기법은 반드시 주석이나 문서로 남겨야 합니다.
